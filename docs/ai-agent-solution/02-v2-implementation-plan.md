## 1. 定位

本文件用于描述在当前已有成熟版本基础上，如何继续演进。

如果团队要做真实版本规划、需求拆解和研发排期，应优先以本文件为准。

## 2. 当前成熟方案复盘

已验证过的主链路：

1. 管理岗输入计划名称
2. 系统基于 ES 做关键字匹配
3. 召回相似历史作战计划
4. 取出历史计划对应的计划配置、商家配置和指标配置
5. 将用户输入、历史数据、商家配置和指标配置一起提供给大模型
6. 大模型输出结构化 JSON
7. 系统解析 JSON，生成推荐的商家筛选结果和指标组
8. 前端回填结果，管理岗修改或直接保存

这个版本已经是一个成熟的计划生成 Copilot。

## 3. 当前版本的主要局限

1. 输入主要是计划名称，表达能力不足
2. 检索主要依赖关键字，召回质量受限
3. 计划生成粒度较粗，缺少分步澄清
4. 商家数据不在本地，必须依赖外部接口和枚举标签
5. 智能化集中在计划创建，后链路价值还没展开

## 4. V2 演进总原则

1. 不推翻现有成熟方案
2. 保留“历史计划增强 + 结构化生成 + 人工确认”的主框架
3. 输入升级为自然语言任务描述
4. 输出升级为计划 DSL + 系统校验
5. 交互升级为可分步、可澄清、可修正
6. 能力逐步扩展到执行、检核和异常

## 5. 演进主线

### 5.1 第一条线：升级计划创建链路

可串联推进的能力：

1. 自然语言计划生成
2. 混合召回
3. 计划 DSL
4. 商家筛选 Copilot
5. 指标配置 Copilot
6. 多轮澄清式配置

这些不是互斥选择，而是一条逐步升级的链路。

### 5.2 第二条线：扩展到任务后链路

1. 执行待办推荐
2. 商家优先级排序
3. 自动检核解释
4. 异常归因
5. 复盘分析

## 6. 关键演进方向

### 6.1 从计划名称补全升级到自然语言计划生成

管理岗可以输入自然语言任务描述，系统先理解计划意图，再召回历史方案并生成结构化计划草案。

详细数据准备见：
[自然语言计划生成的数据准备与存储设计](/Users/zhouzhixiong/code/zuozhanV2/docs/ai-agent-solution/03-nl-plan-data-design.md)

### 6.2 从 ES 关键字匹配升级到混合召回

建议组合：

1. 关键字召回
2. 向量语义召回
3. 指标组合相似召回
4. 商家筛选结构相似召回
5. 历史效果重排

### 6.3 从一次性整包生成升级到两个 Copilot

建议拆成：

1. 商家筛选 Copilot
2. 指标配置 Copilot

商家数据不在本地时，必须采用：

`自然语言 -> 商家查询 DSL -> 外部商家接口`

### 6.4 从一次性生成升级到多轮澄清式配置

当用户输入复杂或不完整时，系统先输出当前理解、缺失信息和待确认项，再逐步收敛。

### 6.5 从计划创建扩展到执行与检核

建议后续扩展：

1. 执行侧 Agent
2. 检核解释 Agent
3. 异常归因 Agent

## 7. 推荐的 V2 架构调整

建议采用以下分层：

1. 计划意图生成层
2. 商家查询代理层
3. 指标配置代理层
4. 计划组装层
5. 审核确认层

这套架构能更好地适配“商家数据依赖外部接口”的现实约束。

## 8. 优先补强能力

优先级最高的几项：

1. 历史计划结构化标签
2. 混合召回和效果重排
3. 计划 DSL
4. 缺失信息识别和澄清式交互
5. 商家筛选自然语言转查询
6. 执行待办推荐和检核解释

## 9. 推荐实施顺序

### 第一阶段：补强当前成熟方案

1. 历史计划结构化标签治理
2. ES 检索升级为混合召回
3. 增加效果重排
4. 引入计划 DSL
5. 优化 JSON 解析与校验链路

阶段目标：
在不改变主交互框架的前提下，提高推荐准确率和生成稳定性。

### 第二阶段：自然语言与分步配置

1. 从计划名称输入升级为自然语言输入
2. 上线缺失信息识别
3. 上线多轮澄清式交互
4. 拆出商家筛选 Copilot
5. 拆出指标配置 Copilot

阶段目标：
把当前“单点补全”升级成“结构化计划构建”。

### 第三阶段：延伸到执行与检核

1. 执行待办推荐
2. 商家优先级排序
3. 自动检核解释
4. 异常归因
5. 复盘分析

阶段目标：
从计划创建智能化，扩展到任务全生命周期智能化。

## 10. 实施结论

最优路径不是推翻重做，而是沿着以下主线演进：

1. 保留现有成熟框架
2. 升级输入、检索和结构化输出
3. 拆分商家和指标两个 Copilot
4. 再逐步扩展到执行、检核和异常处理
