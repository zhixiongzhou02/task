## 1. 召回结果缺乏多样性

### 现象

准备数据的时候，发现存在重复数据，导致召回出来的数据缺乏多样性。

### 影响

1. 检索结果容易被重复样本占满
2. 大模型看到的历史案例重复度过高
3. 结果看起来“很像很准”，但信息增量不高

### 处理方式

1. 计算内容指纹
2. 对相似样本做归类
3. 在召回或重排阶段做去冗余 / 降权处理

### 结论

不要在存储层简单删除相似样本，而是通过：

1. `content_fingerprint`
2. `duplicate_group_id`
3. 召回去冗余
4. 重排降权

来保证多样性和代表性。

## 2. 商家标签和枚举上下文过大

### 现象

一开始把商家的所有 tags 和枚举都给大模型，结果发现内容太大，影响上下文长度和模型效果。

### 影响

1. Prompt 太长
2. 有效信息占比下降
3. 模型注意力被大量无关标签稀释
4. 成本和时延上升

### 处理方式

后续不再直接给全量 tags 和枚举，而是：

1. 先召回相似计划
2. 从相似计划中摘取相关 tag 和枚举
3. 只把与当前意图最相关的边界信息提供给模型

### 结论

上游可用边界不等于都要塞给大模型。

更合理的做法是：

1. 全量边界保留在系统侧
2. 给模型时只裁剪出与当前场景相关的子集

## 3. 索引和样本混在一起的问题

### 现象

一开始索引里直接存入了样本，导致上下文太大，同时影响召回效果。

### 影响

1. 检索索引承载过多非检索字段
2. 检索和样本事实混在一起，字段职责不清
3. 检索结果和上下文组织都变重

### 处理方式

最后把索引和样本分开：

1. 样本表负责存稳定事实
2. 检索索引负责找可召回内容

### 结论

样本层和检索层应分离：

1. 样本表适合作为稳定事实层
2. 检索索引适合作为召回层
3. 两者分开后，检索和上下文组织都会更清晰
