## 1. 召回结果缺乏多样性

### 现象

准备数据的时候，发现存在重复数据，导致召回出来的数据缺乏多样性。

### 影响

1. 检索结果容易被重复样本占满
2. 大模型看到的历史案例重复度过高
3. 结果看起来“很像很准”，但信息增量不高

### 处理方式

1. 计算内容指纹
2. 对相似样本做归类
3. 在召回或重排阶段做去冗余 / 降权处理

### 结论

不要在存储层简单删除相似样本，而是通过：

1. `content_fingerprint`
2. `duplicate_group_id`
3. 召回去冗余
4. 重排降权

来保证多样性和代表性。

## 2. 商家标签和枚举上下文过大

### 现象

一开始把商家的所有 tags 和枚举都给大模型，结果发现内容太大，影响上下文长度和模型效果。

### 影响

1. Prompt 太长
2. 有效信息占比下降
3. 模型注意力被大量无关标签稀释
4. 成本和时延上升

### 处理方式

后续不再直接给全量 tags 和枚举，而是：

1. 先召回相似计划
2. 从相似计划中摘取相关 tag 和枚举
3. 只把与当前意图最相关的边界信息提供给模型

### 结论

上游可用边界不等于都要塞给大模型。

更合理的做法是：

1. 全量边界保留在系统侧
2. 给模型时只裁剪出与当前场景相关的子集

## 3. 索引和样本混在一起的问题

### 现象

一开始索引里直接存入了样本，导致上下文太大，同时影响召回效果。

### 影响

1. 检索索引承载过多非检索字段
2. 检索和样本事实混在一起，字段职责不清
3. 检索结果和上下文组织都变重

### 处理方式

最后把索引和样本分开：

1. 样本表负责存稳定事实
2. 检索索引负责找可召回内容

### 结论

样本层和检索层应分离：

1. 样本表适合作为稳定事实层
2. 检索索引适合作为召回层
3. 两者分开后，检索和上下文组织都会更清晰

## 4. 历史计划不宜过早参与术语归一化

### 现象

在设计自然语言主链路时，曾考虑让历史计划召回结果直接参与“术语归一化”阶段，帮助模型更快判断词义。

### 风险

1. 术语归一化会被方案级历史偏置  
   模型容易按照历史高频方案去理解当前输入，而不是优先按照当前句子本身做词义判断。
2. 词义判断和方案参考会混在一起  
   本来“这个词是什么意思”和“类似场景以前怎么做”是两层问题，过早混合后，中间层会失去纯净性。
3. 可解释性会变差  
   一旦归一化结果同时受到词典候选和历史方案影响，后续很难清楚解释“为什么这个词被归到这个标准术语”。

### 处理方式

最终选择把历史计划召回结果放到归一化之后再使用：

1. 术语归一化优先基于术语词典、商家元数据、指标术语和上下文做判断
2. 历史计划召回结果作为 `similar_plans` 单独产出
3. 在“候选筛选意图”阶段再把 `similar_plans` 作为参考增强输入

### 结论

历史计划更适合回答：

1. 类似场景以前怎么做
2. 以前常用什么筛选条件和指标组合

而不是直接回答：

1. 当前这句话里的词到底是什么意思

更稳的设计是：

1. 先做相对独立的术语归一化
2. 再在后续意图组织或方案生成阶段引入历史计划参考
