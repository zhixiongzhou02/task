## 1. 定位

本文件描述自然语言计划生成的推荐系统链路，重点是“不是单段 Prompt，而是一条可校验、可追踪、可降级的业务流程”。

边界说明：

1. 本文件只回答“跨服务怎么编排”
2. 不重复解释节点一到节点六内部如何做语义理解
3. 不展开每类数据资产如何生产
4. 不展开每个接口字段定义

为什么这些细节要放在本文件：

1. 因为系统上线时最容易出问题的是“服务之间怎么串”，不是概念定义
2. 因为校验点、依赖顺序、降级点天然属于编排层，而不属于单个节点
3. 因为研发排查问题时通常先看时序，再回头看节点细节

## 2. 主链路目标

1. 先理解用户意图
2. 再完成多源召回和主链路推理
3. 再做商家边界映射和 DSL 组装
4. 再校验和降级
5. 最后调用外部商家接口并回填前端

## 3. 推荐主链路步骤

这一节不再重复节点内的语义细节，而是用“服务编排”视角描述主链路。

1. 前端接收用户输入并生成 `request_id`
2. 编排层调用 `candidate-recall-service`，完成短语抽取、多源召回和候选裁剪
3. 编排层调用 `term-normalization-service`，产出 `normalized_terms`
4. 编排层调用 `slot-extraction-service`，产出 `slots`、`missing_slots` 和 `clarifications`
5. 如果缺失关键槽位，则转入澄清模式，不继续往下编排
6. 如果信息足够，则进入 `intent-mapper-service`，生成 `candidate_filter_intent` 并映射到商家服务真实边界
7. 编排层调用 `dsl-builder`，生成最终 `merchant_filter_dsl`
8. 对 DSL 做 Schema 校验和规则校验
9. 校验通过后，再调用商家查询代理和外部商家服务
10. 将商家结果、待确认项、可编辑条件一起回填前端
11. 用户确认后，计划服务保存正式计划

## 4. Mermaid 时序图

```mermaid
sequenceDiagram
    participant U as 管理岗
    participant FE as 前端
    participant NLU as 意图理解服务
    participant TERM as 术语词典服务
    participant RET as 历史计划检索服务
    participant LLM as 计划生成模型
    participant DSL as DSL校验服务
    participant MQA as 商家查询代理
    participant MAPI as 外部商家系统
    participant MGA as 指标配置代理
    participant MC as 指标中心
    participant PLAN as 计划服务

    U->>FE: 输入自然语言计划描述
    FE->>TERM: 术语归一化请求
    TERM-->>FE: 归一化结果
    FE->>NLU: 意图理解请求
    NLU-->>FE: 槽位抽取结果/缺失项
    FE->>RET: 历史计划召回请求
    RET-->>FE: 相似计划样本
    FE->>LLM: 输入用户意图+历史计划+字段字典+指标定义
    LLM-->>FE: 输出计划DSL草案
    FE->>DSL: DSL Schema校验/规则校验
    DSL-->>FE: 校验通过/待修正项
    FE->>MQA: 商家筛选DSL
    MQA->>MAPI: 外部商家查询请求
    MAPI-->>MQA: 商家数量/样例商家
    MQA-->>FE: 商家查询结果
    FE->>MGA: 指标组校验请求
    MGA->>MC: 查询指标定义/模板
    MC-->>MGA: 指标信息
    MGA-->>FE: 合法指标组及解释
    FE->>PLAN: 组装计划草案
    PLAN-->>FE: 返回前端回填结构
    FE-->>U: 展示推荐结果、待确认项、可编辑配置
```

## 5. 关键校验点

1. 术语归一化后，是否存在无法识别的关键业务词
2. 意图理解后，是否缺少关键槽位
3. DSL 生成后，是否满足 Schema
4. 商家查询前，字段和枚举是否合法
5. 指标组生成后，是否存在冲突或不可检核指标
6. 保存正式计划前，是否存在必须人工确认项

## 6. 降级策略

1. 历史计划检索失败：
降级为基于用户输入和基础模板生成

2. 模型生成失败：
降级为推荐历史计划模板，不做自动结构化生成

3. DSL 校验失败：
提示用户修正，不进入商家查询

4. 商家接口失败：
展示条件草案，但不展示商家结果

5. 指标校验失败：
展示候选指标，但标记为待确认

## 7. 实施建议

首期建议按以下顺序落地：

1. 先完成术语归一化、意图理解、历史计划召回
2. 再完成 DSL 生成和校验
3. 再接商家查询代理
4. 再接指标配置代理
5. 最后做多轮澄清和交互优化
