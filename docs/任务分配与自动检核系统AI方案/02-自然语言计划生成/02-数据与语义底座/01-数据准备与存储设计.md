## 3. 历史计划结构化样本

目标：
作为历史经验库，用于关键词召回、向量召回、重排和 few-shot 参考。

为什么要考虑这些细节：

1. 因为没有历史样本，模型只能“凭空生成”
2. 因为历史计划不仅用于召回，还会进入 LLM 上下文和重排
3. 因为后面如果做效果学习，必须先把计划样本结构化

数据来源：

1. 当前已有计划表
2. 历史作战计划记录
3. 历史计划配置快照
4. 历史计划效果数据

推荐字段：

1. 计划 ID
2. 计划名称
3. 计划描述
4. 创建时间
5. 计划周期
6. 业务目标类型
7. 商家筛选条件
8. 指标组合
9. 覆盖商家数
10. 达成率
11. 平均完成周期
12. 是否成功案例
13. 行业标签
14. 区域标签
15. 商家类型标签

推荐存储：

1. 关系表：`plan_history`、`plan_history_tags`、`plan_history_metric_group`、`plan_history_merchant_filter`、`plan_history_effect`
2. AI 检索索引：`idx_plan_history_ai_search`
3. 向量层：历史计划描述、商家筛选摘要、指标组合摘要 embedding

## 4. 商家查询元数据与边界适配

目标：
作为商家查询边界层，把自然语言意图映射到外部商家系统真实支持的字段、标签、枚举和组合限制。

为什么要考虑这些细节：

1. 因为上游能力边界是事实来源，不能让模型自由发明字段
2. 因为同样一句话，最终能不能执行取决于这里的字段、标签、枚举
3. 因为这层也是权限、合法性和可执行性的第一道约束

数据来源：

1. 上游商家服务元数据接口
2. 商家系统字段定义
3. 商家系统标签定义
4. 查询口径约定
5. 区域 / 概念映射规则

推荐字段：

1. 字段编码
2. 字段名称
3. 字段类型
4. 是否支持查询
5. 支持的操作符
6. 枚举值列表
7. 标签值列表
8. 字段说明
9. 同义词
10. 组合限制
11. 是否需要概念展开
12. 概念展开目标类型

推荐存储：

1. 首期可直接复用上游元数据接口 + 本地缓存
2. 适配层缓存：`merchant_meta_cache`、`merchant_meta_adapted`
3. 稳定后可落表：`merchant_filter_field`、`merchant_filter_operator`、`merchant_filter_enum`、`merchant_filter_alias`、`merchant_filter_constraint`
4. 概念映射表：`region_concept_mapping`

## 5. 指标中心标准定义

目标：
作为指标选择的合法边界，支撑指标意图映射、模板召回和冲突校验。

为什么要考虑这些细节：

1. 因为“首单提升”既可能表示业务目标，也可能命中具体指标
2. 因为指标是否可检核、是否可做主指标，都会影响后续计划生成
3. 因为指标中心如果只有定义没有语义扩展，模型就难以稳定选择

推荐字段：

1. 指标编码
2. 指标名称
3. 指标解释
4. 指标分类
5. 指标类型：实时 / 离线
6. 统计周期
7. 是否支持自动检核
8. 是否支持做主指标
9. 是否支持做过程指标
10. 数据来源
11. 口径版本
12. 同义词
13. 适用业务场景

推荐存储：

1. `metric_definition`
2. `metric_version`
3. `metric_category`
4. `metric_alias`
5. `metric_template`
6. `metric_conflict_rule`
7. 可选适配缓存：`metric_term_adapted`

## 6. 业务术语和别名词典

目标：
作为业务语言到系统语言的桥。

为什么要考虑这些细节：

1. 因为管理岗说的是业务话术，不是标准编码
2. 因为术语词典决定了节点一能否先把口语翻译成标准语义
3. 因为这类资产需要长期迭代，不能写死在 Prompt 里

数据来源：

1. 历史计划名称和备注
2. 业务同学日常表达
3. 商家系统与指标系统术语

推荐字段：

1. 术语原文
2. 术语类型
3. 标准编码
4. 标准名称
5. 描述说明
6. 来源系统
7. 置信度
8. 是否启用

推荐存储：

1. 主表：`biz_term_dictionary`
2. 展开表：`biz_term_dictionary_expanded`

### 6.1 为什么术语词典不能只靠 Prompt

术语词典的存在意义，不是给 Prompt 多塞一点背景，而是建立一套可复用、可维护、可追踪的业务语义底座。

为什么要考虑这个细节：

1. 因为管理岗说的是业务话术，不是系统编码。
2. 因为同一个词可能既像业务目标，也像指标术语，必须有标准答案做边界。
3. 因为如果术语只存在于 Prompt 中，后面很难做版本管理、评估和增量更新。

### 6.2 术语词典的生成步骤为什么要单独设计

术语资产不是一次性人工整理完就结束，而是会持续从存量和增量文本中生长出来。

推荐步骤：

1. 候选短语抽取  
   从历史计划名称、描述、备注和真实输入中抽高频短语。
2. 候选归并  
   用规则、相似度和 LLM 辅助把同义表达聚到一起。
3. 标准项确认  
   确认标准名称、编码、类型和可保留别名。
4. 入主表  
   固化标准术语。
5. 展开匹配表  
   生成可在线直接匹配的别名项。

示例：

```json
{
  "term": "首单提升",
  "term_type": "goal_type",
  "normalized_code": "first_order_growth",
  "normalized_name": "首单提升",
  "aliases": ["拉首单", "冲首单", "首单拉升"]
}
```

为什么要考虑这些细节：

1. 因为如果直接把原始短语全部入词典，后面召回会非常脏。
2. 因为术语归并是节点一准确率的前置条件。

### 6.3 为什么要同时准备关键词、短语向量和句子向量

推荐采用三层配合，而不是只依赖单一手段：

1. 关键词 / 别名匹配  
   作用：高精度命中标准术语和稳定别名。
2. 短语向量  
   作用：补召回口语化短语和表达变体。
3. 句子向量  
   作用：利用整句上下文判断整体语义。

为什么要考虑这些细节：

1. 因为只靠关键词，覆盖不了大量口语化表达。
2. 因为只靠向量，召回噪声会明显增大。
3. 因为只有三层组合，系统才有条件做到“高精度 + 可补召回 + 可消歧”。

### 6.4 这些语义资产进入主链路时到底扮演什么角色

节点一在线上直接消费的，不是原始文本池，而是已经准备好的在线可消费资产：

1. `biz_term_dictionary` + `biz_term_dictionary_expanded`
2. `merchant_meta_cache` + `merchant_meta_adapted`
3. `metric_definition` + `metric_alias` + `metric_term_adapted`
4. `region_concept_mapping`
5. `term_vector_index` + `query_sample_vector_index`

这些资产在主链路中的作用分别是：

1. 关键词命中
2. 字段、标签、枚举约束
3. 指标语义命中
4. 概念展开
5. 短语级和句子级语义补召回

## 7. 计划 DSL / JSON Schema

目标：
作为模型输出和系统落地之间的合同。

为什么要考虑这些细节：

1. 因为没有固定 DSL，模型输出无法稳定校验和回填前端
2. 因为 DSL 是“业务理解”到“系统执行”的交界面
3. 因为后续接口、审计、回放都依赖这层结构

首期建议覆盖：

1. 计划元信息
2. 商家筛选条件
3. 指标组
4. 限制条件
5. 待确认项
6. 置信度

示意结构：

```json
{
  "plan_meta": {
    "plan_name": "",
    "goal_type": "",
    "time_range": ""
  },
  "merchant_filters": [],
  "metric_group": [],
  "constraints": [],
  "clarifications": [],
  "confidence": 0.0
}
```

推荐存储：

1. 代码仓库保存 Schema 文件
2. 数据库表保存版本信息：`plan_dsl_schema`、`plan_dsl_version`

## 8. 真实输入语句和人工标注结果

目标：
作为意图理解训练集和评估集。

为什么要考虑这些细节：

1. 因为没有真实输入，就很难知道用户实际怎么说
2. 因为没有评估样本，就只能凭感觉调召回和消歧
3. 因为后续想做弱监督、自动评估和低置信度回流，都要依赖这里

推荐标注内容：

1. 目标类型
2. 商家筛选条件
3. 指标偏好
4. 时间范围
5. 限制条件
6. 缺失项
7. 是否需要澄清
8. 正确 DSL 草案

推荐存储：

1. `nl_plan_query_sample`
2. `nl_plan_query_label`
3. `nl_plan_query_eval_set`

## 9. 推荐存储分层

1. 在线关系库：
存历史计划结构化表、商家边界适配持久化表、指标定义、术语词典、标注样本、DSL 版本

2. Elasticsearch：
存历史计划 AI 检索索引

3. 向量检索层：
存历史计划、业务术语、指标术语和真实输入样本 embedding

4. 代码仓库：
存 DSL Schema、Prompt 模板和规则配置

## 10. 首期最小可行准备建议

1. 100 到 300 条高质量历史计划样本
2. 20 到 30 个常用商家筛选字段
3. 30 到 50 个高频指标
4. 100 到 200 个高频业务术语
5. 一版最小计划 DSL
6. 100 到 200 条人工标注输入样本

## 11. 如何进入系统链路

这六类数据会串成如下链路：

1. 术语词典先做归一化
2. 标注样本帮助意图理解
3. 历史计划做混合召回
4. 商家查询元数据与边界适配生成商家查询边界
5. 指标定义和指标术语生成合法候选指标语义
6. DSL 承接模型输出并做系统校验

为了和后续链路文档保持一致，建议把第 2 到第 6 类数据进一步理解成下面这套在线可消费资产：

1. `biz_term_dictionary` + `biz_term_dictionary_expanded`
2. `merchant_meta_cache` + `merchant_meta_adapted`
3. `metric_definition` + `metric_alias` + `metric_term_adapted`
4. `region_concept_mapping`
5. `nl_plan_query_sample` + `nl_plan_query_label`

也就是说：

- `01-数据准备与存储设计` 负责回答“这些资产从哪来、怎么存”
