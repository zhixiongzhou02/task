## 1. 为什么需要这份策略

即使系统已经提供了：

1. 术语词典
2. 商家字段、标签、枚举元数据
3. 历史计划参考
4. 指标定义

模型仍然可能出现以下问题：

1. 输出了不存在的字段名
2. 输出了业务话术，而不是系统可执行枚举
3. 把区域概念当成可直接执行值
4. 对同一个词给出多个都“看起来合理”的解释
5. 在信息不够完整时，假装已经可以生成最终 DSL

所以，这件事不能只靠 Prompt 约束，而要靠一套完整的：

1. 歧义判断机制
2. 自动修正机制
3. 澄清机制
4. 错例回流机制

## 2. 先定义几个核心概念

### 2.1 什么叫“乱输出”

乱输出不一定是完全胡说八道，更常见的是：

1. 语义方向对，但字段名错
2. 业务表达对，但枚举值不合法
3. 结构看起来完整，但实际上不可执行
4. 历史参考被误当成当前事实

### 2.2 什么叫“高歧义”

高歧义不是“两个词相似”，而是：

**同一个输入片段，对应多个都像正确答案的候选解释，系统无法安全拍板。**

例如：

- `第一单`
  可能对应：
  - `goal_type = first_order_growth`
  - `metric_alias = first_order_cnt`

- `活跃`
  可能对应：
  - 活跃商家数
  - 活跃率
  - 活跃提升目标

### 2.3 什么叫“业务语义不明确”

不是词义难判断，而是：

**信息不够完整，即使词义理解没问题，也不足以继续安全生成 DSL。**

例如：

- `做一个首单提升计划`

这里“首单提升”本身未必歧义，但缺少：

1. 区域
2. 行业
3. 商家范围
4. 排除条件

## 3. 可自动修正，具体怎么做

推荐由系统规则层提供一个标准化修正模块，例如：

- `output-normalizer`
- 或 `intent-mapper`

其职责是：

1. 接收模型草案
2. 尝试把“业务表达”修正成“系统标准表达”
3. 对无法修正的内容明确报错或转入澄清

### 3.1 字段别名修正

#### 场景

模型输出：

```json
{
  "field": "industry",
  "operator": "=",
  "value": "餐饮"
}
```

但系统真实字段是：

- `industry_code`

#### 建议实现

准备一张字段别名映射表，例如：

```json
{
  "industry": "industry_code",
  "region": "region_code",
  "city": "region_code",
  "merchant_stage": "tag_code"
}
```

处理逻辑：

1. 先判断字段是否是合法字段
2. 如果不是，查字段别名字典
3. 如果命中，替换成标准字段
4. 如果没命中，标记为非法字段

#### 结论

字段别名修正，长期应该是**工程实现**。

如果冷启动时还没有字典，可以用 LLM 辅助发现候选别名，但线上运行不应长期依赖 LLM 临场猜字段。

### 3.2 枚举别名修正

#### 场景

模型输出：

```json
{
  "field": "potential_level",
  "operator": "in",
  "value": ["高潜"]
}
```

但系统真实枚举可能是：

- `A`
- `B`
- `C`

同时业务定义为：

- `高潜 -> [A, B]`

#### 建议实现

准备按字段拆分的枚举别名映射，例如：

```json
{
  "potential_level": {
    "高潜": ["A", "B"],
    "中潜": ["C"]
  }
}
```

处理逻辑：

1. 检查 value 是否在合法枚举中
2. 如果不在，查别名映射
3. 命中后展开成标准值
4. 未命中则判非法枚举

#### 结论

枚举别名修正也应该是**工程实现**。  
LLM 只适合冷启动时辅助归并，不适合线上每次临场判断。

### 3.3 区域概念展开

#### 场景

模型输出：

```json
{
  "field": "region_code",
  "operator": "=",
  "value": "华东"
}
```

但商家服务真实只支持城市级枚举：

- 上海
- 杭州
- 苏州
- 南京

#### 建议实现

准备概念映射表，例如：

```json
{
  "华东": ["上海", "杭州", "苏州", "南京"],
  "长三角": ["上海", "杭州", "苏州", "宁波"]
}
```

处理逻辑：

1. 识别 `华东` 是概念词
2. 查概念映射表
3. 把：
   - `operator "="`
   替换成：
   - `operator "in"`
4. 把：
   - `value "华东"`
   替换成：
   - `value ["上海", "杭州", "苏州", "南京"]`

#### 结论

区域概念的“识别”可以有 LLM 参与，  
但区域概念的“展开”长期必须是**工程实现**，不能交给模型每次临场猜。

## 4. 现有别名与映射资产如何复用，缺口如何补齐

当前项目的前提不是“完全没有字典”，而是：

**已经存在一批可以直接复用的别名和映射资产，重点在于怎么用好它们，并把在线错误继续补回去。**

### 4.1 当前可直接复用的资产

按当前方案设计，至少已经有这些可承接的资产：

1. 商家字段/标签别名  
   例如：
   - `merchant_filter_field`
   - `merchant_filter_tag`
   - `merchant_meta_adapted_cache`

2. 指标别名  
   例如：
   - `metric_alias`

3. 业务术语别名  
   例如：
   - `biz_term_dictionary`
   - `biz_term_dictionary_expanded`

4. 区域概念映射  
   例如：
   - `region_concept_mapping`

### 4.2 当前阶段的重点不是“从零造字典”

更贴近实际的重点是：

1. 先复用已有别名和映射资产
2. 在在线处理时发现缺口
3. 把缺口沉淀成新的别名、映射或规则

也就是说，当前阶段更适合做：

- 复用 + 补齐

而不是：

- 从零发现 + 从零建设

### 4.3 线上如何复用已有资产

线上处理时，建议顺序如下：

1. 先查现有业务术语词典  
   解决：
   - `拉首单`
   - `冲首单`
   - `第一单拉升`
   这类表达如何映射到标准术语

2. 再查现有商家元数据适配层  
   解决：
   - `新店`
   - `关店商家`
   - `行业`
   - `城市`
   这类表达如何映射到字段、标签和枚举

3. 再查现有指标别名  
   解决：
   - `首单`
   - `转化率`
   - `GMV`
   这类表达如何映射到指标定义

4. 最后查现有概念映射  
   解决：
   - `华东`
   - `长三角`
   这类概念如何展开成系统可执行值

### 4.4 如果现有资产里没有命中怎么办

这时不要直接判“系统不支持”，而要继续分两层处理：

#### 第一层：在线处理

先判断当前缺口属于哪一类：

1. 可以自动修正  
   例如：
   - 只是缺一个别名映射
   - 只是缺一个概念展开

2. 需要澄清  
   例如：
   - 有多个候选解释
   - 当前系统无法安全拍板

3. 必须拒绝  
   例如：
   - 不存在合法字段
   - 不存在合法标签
   - 存在冲突条件

#### 第二层：离线修复

把这次缺口回流成：

1. 新别名
2. 新概念映射
3. 新高歧义词规则
4. 新样本

### 4.5 缺口补齐的推荐路径

推荐按下面顺序补：

1. **优先补映射资产**
   例如：
   - 标签别名
   - 字段别名
   - 区域概念映射

2. **再补样本资产**
   例如：
   - 高歧义短语样本
   - 错误消歧样本

3. **最后再调策略**
   例如：
   - TopN 阈值
   - 高歧义判定规则
   - 澄清触发条件

### 4.6 核心原则

当前阶段更准确的原则不是：

- “LLM 负责发现候选，工程负责固化规则”

而是：

- **已有资产先复用**
- **在线缺口再发现**
- **发现后的缺口持续回流到资产和规则**

## 5. 高歧义怎么判断

高歧义的判断，推荐由系统规则层先做，不要把第一步判断直接交给 LLM。

### 5.1 规则一：前两名候选分差太小

例如：

- top1 = `0.86`
- top2 = `0.855`

分差：

- `0.005`

如果小于阈值（例如 `0.08`），就可判为高歧义。

### 5.2 规则二：候选跨类型

例如：

- 一个候选是 `goal_type`
- 一个候选是 `metric_alias`

即使分差不算特别小，也属于高风险，因为说明模型可能在“目标”和“指标”之间摇摆。

### 5.3 规则三：命中高歧义词清单

例如：

- `首单`
- `第一单`
- `活跃`
- `转化`

这些词天然语义宽，容易落到多个系统定义上。

### 5.4 规则四：多来源给出不同解释

例如：

1. 业务术语词典偏向 `goal_type`
2. 指标术语偏向 `metric`
3. 句子向量又召回另一类样本

这说明候选冲突较大，也应该提高歧义等级。

### 5.5 一个简单的判断公式

```text
is_high_ambiguity =
  (top1_score - top2_score < 0.08)
  OR (type1 != type2)
  OR (raw_phrase in ambiguity_lexicon)
  OR (multi_source_conflict = true)
```

## 6. 业务语义不明确怎么判断

这和高歧义不是一回事。

高歧义是：
- 候选很多，分不清选哪个

业务语义不明确是：
- 信息不够全，即使理解了词，也没法安全生成 DSL

### 6.1 规则一：关键槽位缺失

例如一个可执行草案至少需要：

1. `goal_type`
2. `region / scope`
3. `industry / merchant_group`

如果缺少这些关键槽位中的任意一部分，就不能直接进入最终 DSL。

### 6.2 规则二：无法落成可执行条件

例如：

- 用户只说“高潜商家”
- 但系统当前没有能映射的标签、字段或概念映射

这时即使词义被理解了，也不能直接执行。

### 6.3 规则三：需要澄清的问题过多

如果系统判断当前至少还要补 2 到 3 个关键问题，说明原输入过于模糊。

此时更合适的处理是：

1. 进入澄清式多轮
2. 而不是强行生成半残缺草案

### 6.4 一个简单判断公式

```text
is_semantically_incomplete =
  (missing_required_slots_count > 0)
  OR (mappable_conditions_count == 0)
  OR (clarifications_count >= 2)
```

## 7. 高歧义出现后，谁来区分哪个才是对的

推荐分两层：

### 第一层：系统先判“难不难”

系统负责：

1. 召回候选
2. 打分
3. 判断是否高歧义

### 第二层：LLM 再判“更像哪个”

只有当系统已经认定：

- 当前短语高歧义

才把：

1. 原始输入
2. 候选集合
3. 候选来源
4. 当前上下文

交给 LLM 做消歧。

#### 核心关系

- 系统负责：判难度
- LLM 负责：在可控候选里做最终判断

## 8. 如果 LLM 已经理解错了，下一次怎么知道哪个是对的

这个问题的关键不在于“再问一次同一个模型”，而在于要有**反馈闭环**。

### 8.1 记录错例

每次错误都应该记录：

1. 原始输入
2. 候选集合
3. 模型最终选择
4. 人工确认或系统最终正确答案

### 8.2 回流成系统资产

把这些错例回流成：

1. 术语词典补充
2. 高歧义词清单
3. 枚举别名表
4. 概念映射表
5. 标注样本

### 8.3 对总是容易错的场景，改成澄清而不是硬猜

例如如果：

- `第一单`

经常在：

1. `goal_type`
2. `metric_alias`

之间摇摆，就不要长期让模型硬猜，而应该改成：

- 进入澄清问题
- 或加强候选约束

#### 核心原则

**下一次变对，不是靠模型突然变聪明，而是靠系统把错例沉淀成规则和资产。**

## 9. 建议的处理分流

### 9.1 能自动修正

例如：

1. 字段别名
2. 枚举别名
3. 区域概念展开

处理方式：

- 自动改正后继续往下走

### 9.2 候选高歧义

例如：

- `第一单`

处理方式：

1. 保留 TopN 候选
2. 交给 LLM 消歧
3. 如仍不稳定，则进入澄清

### 9.3 业务语义不明确

例如：

- `做一个首单提升计划`

处理方式：

1. 不直接生成最终 DSL
2. 进入澄清问题

### 9.4 高风险非法输出

例如：

1. 不存在的字段
2. 不存在的标签
3. 包含和排除条件冲突

处理方式：

1. 拒绝执行
2. 返回错误或澄清

## 10. 一句话总结

这套策略的核心不是“让模型永远不出错”，而是：

1. 能修正的，优先自动修正
2. 歧义高的，让系统先判断，再由 LLM 消歧
3. 信息不够的，转入澄清
4. 错过的，沉淀成字典、规则和样本

这样模型即使偶尔理解错，也不会直接把错误带进最终 DSL。
